---
name: writing-epics
description: How to write epic tasks or job stories. Use when you are writing a task 
---

- Write tickets for both business and engineering audiences; avoid jargon-only or business-only framing.

- Use a consistent template across tickets to improve findability and reduce confusion.

- Include enough information to prevent “just do it” ambiguity; upfront clarity saves rework.

- Start with a job story in “WHEN / I WANT TO / SO I CAN” form; focus on user value, not implementation.

- Keep stories sprint-sized; break down into smallest reasonable deliverables.

- Prefer vertical slices that deliver end-to-end value each sprint, not horizontal layer-by-layer work.

- Capture acceptance criteria as a checklist, ideally in GIVEN/WHEN/THEN format.

- Build acceptance criteria with input from the whole team (design, engineering, performance, architecture).

- Use acceptance criteria as the sign-off baseline and, when possible, map to automated tests.

- Attach wireframes/mockups to define visual expectations that acceptance criteria can’t convey.
